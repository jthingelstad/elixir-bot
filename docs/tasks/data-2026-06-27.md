# Data Brief 2026-06-27

## Summary

No new first-seen CR API schema path, event tag, card catalog payload, or battle game mode appeared since the last Data Analyst run. The actionable finding is a derived-data integrity bug: `detections.occurred_at` mixes compact Clash Royale timestamps and ISO timestamps, while recent-event read tools assume one lexical format.

Issue: https://github.com/jthingelstad/elixir-bot/issues/142

## Evidence

- Last-run boundary: `2026-06-27T03:01:40.880Z`.
- `api_sentinel_observations` had no rows with `first_seen_at >= '2026-06-27T03:01:40'`.
- Retained `raw_api_payloads` since the boundary: `player=196`, `player_battlelog=125`, `clan=38`, `currentriverrace=16`, and no new retained `events` or `cards` payload hash.
- `game_mode_contexts` still shows the same active event set last seen on June 27: Restless Undead, Mega Draft Challenge, Classic Challenge, Grand Challenge, Merge Tactics, Classic 1v1, Classic 2v2, and Seasonal Trophy Road.
- Restless Undead and event-mode activity were already handed off in issue #105 and downstream PM/build issues, so this run did not open a duplicate product issue.

## Integrity Finding

`detections.occurred_at` has two live timestamp shapes:

- `compact_cr`: 501 rows across 3 detection types, from `20260412T191930.000Z` to `20260628T025344.000Z`.
- `iso`: 457 rows across 17 detection types, from `2026-06-07T07:47:00Z` to `2026-06-28T03:03:08.317969+00:00`.

The compact rows are all battle-derived signals:

- `battle_trophy_push`: 249 rows.
- `battle_hot_streak`: 246 rows.
- `ranked_activity_pulse`: 6 rows.

The ISO rows include the milestone, badge, roster, war, cohort, and progression detections that Elixir uses for clan memory and member-facing recognition.

## Impact

A parsed timestamp window since the last run finds 55 true detections:

- `battle_trophy_push`: 20
- `best_trophies_peak`: 14
- `badge_earned`: 7
- `card_level_milestone`: 4
- `collection_level_milestone`: 2
- `cohort_wave`: 2
- `new_card_unlocked`: 2
- `new_champion_unlocked`: 2
- `path_of_legend_promotion`: 1
- `war_update`: 1

The current CR-string cutoff pattern (`occurred_at >= '20260627T030140'`) returns only 20 rows, all `battle_trophy_push`.

The seven-day window shows the same distortion: parsed timestamps find 273 detections across 17 types, while the current CR-string read path returns 122 detections across only `battle_trophy_push`, `battle_hot_streak`, and `ranked_activity_pulse`.

## Downstream State

The likely affected read surfaces are:

- `event_core/read/event_facades.py`, where the module comments say `detections.occurred_at` is CR-format and `summarize_event_windows()` / `list_recent_events()` compare it with CR-format cutoffs.
- `agent/tool_exec.py` `get_elixir_state`, which uses those facades for `event_summary`, `recent_events`, and default state.
- `runtime/helpers/_reports.py`, `runtime/jobs/_core.py`, and `runtime/jobs/_memory.py`, which pull recent event context for reports, runtime context, and memory synthesis.
- `event_core/read/tools.py::get_player_detections()`, which orders by raw `occurred_at DESC` and can misorder per-player history when formats differ.

## Product Intelligence

This is not a new Product Manager capability seed; it is a read-side data-quality defect. It can make Elixir's recent-event context look like the clan mostly produced trophy-push signals while omitting same-window badges, card milestones, Path of Legends promotions, war updates, cohort waves, and unlocks.

The Build Manager should normalize detection timestamps at write time or add a read-side normalization/backfill path so all detection windows and ordering use one comparable UTC representation.
